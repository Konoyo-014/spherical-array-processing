## 我直接修掉的点

第一处是真 bug，落在 `spherical_array_processing/ambi/uhj.py:75`。原实现对 FIR Hilbert 走的是 `np.convolve(signal, fir, mode="same")`。这个写法只有在 **输入长度不小于 FIR 长度** 时，输出长度才等于输入长度；一旦 `fir_taps > len(signal)`，NumPy 会返回长度为较长那个数组的结果，也就是 **返回 `fir_taps` 长度**。这和公开 API 声称的“输出与输入同长”不一致，也会同时影响 `uhj_encode` 和 `uhj_decode` 的短信号路径。我把这里改成了 **先做 full convolution，再按群延迟 `(N-1)/2` 居中裁剪**，这样无论输入多短，输出都严格保持原长度，并且内部样本的时间对齐逻辑仍然成立。对应的回归测试加在 `tests/test_ambi_uhj.py:139`。

第二处是 `Data.Delay` 的结构校验，落在 `spherical_array_processing/hrtf/sofa.py:164`。原实现只要看到二维数组且第二维是 `2`，就会直接当成 `(M, 2)` 接受，所以像 `(3, 2)` 配 `M=5` 这种 **方向数不匹配** 的非法 SOFA 文件，也会被静默读进来。这个问题会让 `HRTFDataset.data_delay_samples` 进入一个和 `source_grid` 不一致的状态。我现在把接受条件收紧成三种：**`(2,)`、`(1, 2)`、`(M, 2)`**。非零但不合法的形状会直接抛 `ValueError`，而 **全零块仍然折叠为 `None`**，因为数据类本身就是用 `None` 表示“没有预延迟”。对应回归测试加在 `tests/test_hrtf_sofa.py:127`。顺手我也把过期文档修正到了 `spherical_array_processing/hrtf/sofa.py:239`，现在不会再误写成 `Data.Delay` 永远是零的 `(1, 2)` 块。

第三处不是实现 bug，而是物理表述需要收紧，落在 `spherical_array_processing/ambi/translation.py:20` 和 `spherical_array_processing/ambi/translation.py:150`。原文把低 `kr` 条件下的 PWD 平移写成了“exact”。这在严格意义上并不成立，因为从 **仅有四个 FOA 系数** 反推出的平面波分布，本质上是一个 **一阶角向点扩散函数**，不是方向上的 delta。于是零平移时它当然恢复恒等，但只要 `r ≠ 0`，哪怕是单个平面波，重建出来的平面波集合也会在不同方向上拿到不同的相移，误差会随着 `kr` 增大而上升。这里我把文档改成了更准确的说法：它在 **`|kr| ≪ 1` 时复现主导几何相位项**，但不是严格精确平移；FFT 零填充也只能 **减弱** 周期延拓缝合带来的污染，不能从数学上彻底消灭分数延迟的周期性伪影。

## 关于平移数学本身

你最关心的相位符号，我确认是 **正号**。这里的 `û` 不是物理波矢方向，而是代码库一贯使用的 **源方向 / DOA 方向**。对于来自 `+x` 的远场平面波，真实传播方向其实是 `-x`。如果听者向 `+x` 也就是朝源方向移动 `r`，到达时间会减少 `τ = û·r/c`，于是新位置的信号是 `x(t + τ)`，在频域里正好对应 `X(f) exp(+j 2π f τ)`。这一点和当前实现 `spherical_array_processing/ambi/translation.py:160` 完全一致，而且也和现有符号测试吻合。

你问到的 **`Y.T @ pinv(Y.T) = I_4`**，在默认 `n_decomposition_dirs=32` 的 Fibonacci 网格上数值上成立，而且条件数非常好。我额外做了数值检查，`M=32` 时 `cond(Y.T)` 约为 `1.015`，`max(abs(Y.T @ pinv(Y.T) - I))` 在 `1e-14` 量级。这也解释了为什么 `test_zero_translation_is_identity` 能稳定压到 `1e-10` 甚至更低。这里成立的关键不是“网格近似积分得好”，而是 **`Y.T` 作为 `4 × M` 矩阵满行秩**，所以 Moore–Penrose 逆会在这四维行空间上给出精确恒等。

更值得讲清的是 **“平移一个 plane-wave FOA 是否有物理意义”**。答案是有，但语义要说准。对于真正的远场平面波，平移并不会引入“距离”这个近场参数，因为远场本来就没有有限源距；它只会引入 **沿 DOA 方向投影的传播时差**。所以你现在的测试，去验证 `û·r/c` 这个几何预测是否被复现，是对的。不过当前算法不是在做“已知单平面波的精确传播”，它做的是“先用一阶 PWD 核把 FOA 解释成一簇方向分量，再给这些方向分量施加相移”。因此它对 **到达时间的主导项** 预测是可信的，对 **完整波形** 的精确性则要保守表述。你现在测试里用 `200 Hz`、`10 cm` 这种低 `kr` 条件，只检查符号、亚采样级到达时间，以及横向位移下失真受控，这个容差设计是合理的。

至于 `nfft = max(next_pow2(N), 2N)` 这一档零填充是否“足以避免 circular wraparound artefacts”，严格回答是否定的。**任何有限长度 DFT 上的分数延迟相位乘法，本质上都还是周期延拓模型**。只要不是整数采样位移，就不存在一个有限 `nfft` 能把时域支撑严格截断到原窗口内。现在这段做法的作用，是把周期拼接缝尽量推到被裁掉的区域，让常见的短平移、带窗信号在保留区间内误差够小。这个实现选择是工程上合理的，但文档里不能把它写成“彻底避免”。我已经把这句话改正了，位置在 `spherical_array_processing/ambi/translation.py:150`。

## 关于 FIR Hilbert

你给出的理想脉冲响应公式 **是对的**。`h[n] = (1 - cos(πn')) / (πn')` 在 `n' ≠ 0` 时，偶数样点正好为零，奇数样点正好化成经典的 **`2 / (πn')`**，所以它就是离散时间理想 Hilbert 变换器的 Type-III 形式。当前实现 `spherical_array_processing/ambi/uhj.py:55` 在这点上没有数学错误。

关于 **群延迟 `(N-1)/2`** 和时间对齐，你的物理判断本身是对的，但原来的实现方式并不普适。线性相位奇长度 FIR 的确有固定群延迟；如果从 full convolution 中 **对称裁掉 `(N-1)/2` 个样本**，保留下来的那一段就和输入在内部对齐。问题只在于 `np.convolve(..., mode='same')` 并不总是执行“按第一个输入长度裁剪”这件事，所以 আগে 那个实现只在长信号时碰巧成立。现在修成显式 full-convolution 居中裁剪之后，这个论断才真正无条件成立。

至于 **FIR Hilbert 是否会随 tap 数增加而收敛到 FFT Hilbert**，答案也是肯定的，但要加一个技术限定：它收敛的是 **远离边界的内部样本**，因为 FFT Hilbert 默认是整块周期信号模型，而有限长 FIR 在边界上一定会看到截断效应。你现在的回归测试 `tests/test_ambi_uhj.py:119` 用 interior margin 来比较两者，并给 `513` taps 留出 `< 5e-3` 的容差，这个设计是合理的。

## 关于 SOFA `Data.Delay`

这块主路径现在是对的。读入时对 **`(1, 2)` 和 `(M, 2)`** 的处理语义正确，写出时对 `HRTFDataset.data_delay_samples` 的 **`(2,)` 和 `(M, 2)`** 检测也正确，相关验证在 `spherical_array_processing/hrtf/sofa.py:345` 这一段是干净的。经过这次修正以后，读路径也不会再把方向数不匹配的 `Data.Delay` 悄悄吞下。

你问到 **“全零块读成 `None` 是否合适”**，我的判断是 **合适，但它是一个 API 设计取舍**。从信号语义看，全零 `Data.Delay` 和缺失 `Data.Delay` 没有区别，都会表示“没有预延迟”；而数据类也已经把 `None` 定义成这个语义，所以读路径折叠成 `None` 是一致的。代价在于，你无法保留“原文件里是否显式写了一个全零 `(M, 2)` 块”这种纯表示层信息。这个损失不影响渲染正确性，但如果你未来想做 **字节级或结构级 round-trip 保真**，那就需要把“缺失”和“显式零块”区分开来，可能要在 `metadata` 里额外记一个标志。
