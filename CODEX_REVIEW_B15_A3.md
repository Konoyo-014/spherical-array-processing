# B15 A3 Review

**Verdict: TAG**

这次 A3 的核心目标，也就是把 **first-order directional capsules** 从底层声学径向系数一路贯通到 **稳定公开 API**，现在已经成立了。我复核到的稳定公开链路是 `ArrayGeometry` 的类型层声明、`spherical_array_processing.acoustics.radial` 里的 `plane_wave_radial_bn` / `bn_matrix` / `sph_modal_coeffs`、`spherical_array_processing.array.simulation.simulate_sh_array_response`，以及 `spherical_array_processing.encoding.radial_filters` 里的 `radial_equalizer_tikhonov`、`radial_equalizer_wng_limited` 和统一入口 `radial_equalizer`。在 `encoding` 里，`dir_coeff` 已经从 equalizer 层显式透传到 `bn_matrix`，对应实现位于 `spherical_array_processing/encoding/radial_filters.py:16`、`spherical_array_processing/encoding/radial_filters.py:42`、`spherical_array_processing/encoding/radial_filters.py:103` 和 `spherical_array_processing/encoding/radial_filters.py:177`。在声学层，`dir_coeff` 已经贯穿 `plane_wave_radial_bn`、`bn_matrix` 和 `sph_modal_coeffs`，对应位置是 `spherical_array_processing/acoustics/radial.py:172`、`spherical_array_processing/acoustics/radial.py:286` 和 `spherical_array_processing/acoustics/radial.py:350`。我也额外检索了 `encoding` 其余模块、`decoding`、`beamforming` 和 `diffuseness`，没有发现其他稳定公开函数既接受 `array_type` 又应该继续向 `bn_matrix` / `plane_wave_radial_bn` / `sph_modal_coeffs` 透传 `dir_coeff` 却尚未透传的情况。

数值极限这部分我重新跑过。`simulate_sh_array_response` 在 `α=0.5` 时与 `cardioid` 的差值是 **max|Δ| = 0.0**，在 `α=1.0` 时与 `open` 的差值也是 **max|Δ| = 0.0**。`α=0.0` 时，DC bin 的闭式结果就是 **`cos γ`**，我复核到单麦克风正前 / 正后两个方向分别得到 **`+1`** 和 **`-1`**，没有任何机器精度残差。equalizer 侧我分别重跑了 `radial_equalizer_tikhonov` 和 `radial_equalizer_wng_limited` 的 `α=0.5 ≡ cardioid`，以及统一入口 `radial_equalizer` 的 `α=1.0 ≡ open`，结果同样都是 **max|Δ| = 0.0**。这说明你现在的实现不是“数值上接近”，而是由于公式在代码路径上真正收敛到同一代数表达式，所以得到了 **bitwise-identical** 的结果。

校验严格性也基本到位。`radial_equalizer` 家族现在会在 equalizer 层直接拒绝 `array_type="directional"` 但缺少 `dir_coeff` 的调用，并且会直接拒绝 `array_type="rigid", dir_coeff=0.5` 这种本来很容易被静默吞掉的误用，错误消息来自 `spherical_array_processing/encoding/radial_filters.py:16` 这层，而不是等到底层 `bn_matrix` 才报。这一点与你的目标一致。我手动复核时，`radial_equalizer(2, kr, array_type="rigid", dir_coeff=0.5)` 会抛出清晰的 `ValueError`，`radial_equalizer(2, kr, array_type="directional")` 也会在 equalizer 层抛出清晰的 `ValueError`。

真正存在问题的是 `ArrayGeometry` 的 `metadata` 文档契约。你原先把 `metadata["dir_coeff"]` 写成了会被 simulation / encoding 层读取，但当前稳定 API 实际并不会自动读取这个字段。代码库里我没有找到任何下游对 `geometry.metadata["dir_coeff"]` 或 `geometry.metadata.get("dir_coeff")` 的读取。为了避免发布一个 **文档上承诺、运行时却不兑现** 的假契约，我已经把说明收紧成“这是一个 advisory convention，当前稳定 simulation / encoding API 仍然要求显式传 `dir_coeff`”。这个修正位于 `spherical_array_processing/types.py:116`。现在文档和实现是一致的。

还有一个我建议你记在后续小单里的 **ergonomics** 点，但它不构成这次 A3 的阻塞项。`simulate_sh_array_response` 目前会在 `array_type="directional"` 且缺少 `dir_coeff` 时立刻报错，这很好，对应 `spherical_array_processing/array/simulation.py:197`。但是它还没有像 equalizer 层那样拒绝“非 directional 却传了 `dir_coeff`”的误用，而且越界 `dir_coeff` 的错误也仍然来自更底层的声学函数，而不是 simulation 层自己做完整前置校验。这不会破坏数值正确性，因为底层最终还是会兜住越界值，但在 API 体验上，它现在没有 equalizer 家族那么严格、那么对称。

如果把范围扩大到 **非稳定公开 API**，我确实还看到一些开发层残留没有完全跟上 directional plumbing。`spherical_array_processing/repro/politis/functions.py:83` 的 `sph_array_noise` 和 `spherical_array_processing/repro/politis/functions.py:100` 的 `sph_array_noise_threshold` 都接受 `array_type`，但没有 `dir_coeff` 入口，因此不能完整表达 generic directional capsule。更明显的是 `spherical_array_processing/repro/politis/functions.py:477` 的 `getDiffCohMtxTheory` 虽然签名里已经有 `dir_coeff`，但在 `spherical_array_processing/repro/politis/functions.py:487` 调 `sph_modal_coeffs` 时并没有把它传下去。这些地方属于 `repro` 开发层；按照顶层包说明，它们不属于稳定公开 API，所以我没有把它们算作 A3 的阻塞问题，但如果你想把开发层也整理干净，它们是下一批最直接的修补点。
